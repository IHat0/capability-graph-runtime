# Pulsate preset run API

## Scope

The versioned Pulsate API coordinates a discovered quantum-preflight manifest through the repository's existing trusted runner. It does not reimplement problem construction, Hamiltonian construction, exact diagonalization, VQE, verification, authorization, or receipt generation. `ExistingQuantumPreflightExecutor` calls `run_trusted_reference` and projects its canonical artifacts into typed browser responses.

This is a bounded, single-process development coordinator, not a production-distributed queue. Dynamic intake compiles only its bounded scientific language; it does not invoke Qwen repair. IBM Quantum execution is an explicitly configured extension and never changes the local-simulator path.

```text
discovered preset -> trusted ManifestEnvelope -> durable run envelope
  -> ExistingQuantumPreflightExecutor -> authoritative runner artifacts
  -> typed results, verification, and receipt responses
```

Preset, experiment, structure, Hamiltonian, run, receipt, and execution-environment identities remain separate. Internal filesystem paths are never returned.

## API and status

- `GET /api/v1/health` and `GET /api/v1/runs/capability` expose local and IBM capability separately.
- `POST /api/v1/runs` accepts a server-resolved preset or dynamic experiment identifier and its exact planned execution target, then returns `202`.
- `GET /api/v1/runs/{run_identifier}` returns durable status and identity.
- `/results`, `/verification`, and `/receipt` return strict typed projections of authoritative evidence. The public receipt exposes stable identities and authorization, while the complete canonical receipt remains server-controlled evidence. Before evidence exists they return `409` with a typed error.

The browser cannot supply paths, commands, executables, arguments, result roots, images, or code. Unknown fields are rejected. Run identifiers and directories are generated by the server. An `Idempotency-Key` of 8–128 identifier characters deduplicates an identical request; conflicting reuse returns `409`.

Observable local states remain `queued`, `validating`, `running_quantum_workflow`, `authorized`, `rejected`, `failed`, and `interrupted`. IBM runs additionally publish `running_local_preflight`, `awaiting_ibm_submission`, `queued_on_ibm`, `running_on_ibm`, and `verifying_ibm_result`. The terminal states remain unchanged. Rejection is a completed scientific decision; failure is an execution or artifact-processing error.

## Persistence and recovery

The default root is ignored `.pulsate-runs/`. Each server-named run directory contains atomic request/state JSON and the required summary, results, verification, receipt, or sanitized error. The nested quantum-preflight artifact directory remains authoritative.

Coordinator construction and module import do not create directories, inspect quantum dependencies, recover runs, acquire resources, or start worker threads. FastAPI lifespan explicitly starts and closes the coordinator. Startup acquires an exclusive run-root ownership lock, validates configuration and execution prerequisites, and only then recovers persisted state. A second live coordinator cannot own the same run root.

On startup, terminal evidence is preserved, idempotency records are restored, and active runs become `interrupted`; calculations are never silently restarted or deleted. Shutdown stops admission before waiting for bounded workers and releasing root ownership. State is lock protected and JSON replacement is atomic. Concurrency defaults to one and is clamped to four.

Before projection, every required runner artifact must be a regular, non-symlink file directly inside the controlled result directory and within the manifest result-size policy. The receipt, result artifacts, verification report, warnings, manifest, summary, and authorization decisions are model-validated and cross-linked by recomputed identities. Inconsistent evidence fails the run and cannot authorize through the browser API.

## Configuration and development

- `PULSATE_EXECUTION_ENABLED`: `true` enables local execution; default `false`.
- `PULSATE_RUN_ROOT`: durable root; default `<repository>/.pulsate-runs`.
- `PULSATE_MAX_CONCURRENT_RUNS`: 1–4 workers; default `1`.
- `PULSATE_MAX_RUN_SECONDS`: server execution ceiling from `1` through `3600`; default `180`. The trusted runner receives the lower of this value and the manifest execution-policy limit.
- `PULSATE_QUANTUM_IMAGE_IDENTIFIER`: environment evidence identifier; default `local-uncontainerized`.
- `PULSATE_IBM_QUANTUM_TOKEN`, `PULSATE_IBM_QUANTUM_INSTANCE`, and `PULSATE_IBM_QUANTUM_BACKEND`: server-only IBM configuration; all are required for v1.
- `PULSATE_IBM_TARGET_PRECISION`, `PULSATE_IBM_OPTIMIZATION_LEVEL`, `PULSATE_IBM_SEED_TRANSPILER`, and `PULSATE_IBM_MAXIMUM_SECONDS`: optional bounded server policies; defaults are `0.015`, `2`, `7341`, and `1800`.
- `PULSATE_IBM_PREFLIGHT_HANDOFF_ROOT`: controlled request/handoff exchange used by the live OS-isolated, no-network preflight coordinator.
- `PULSATE_IBM_SCIENTIFIC_IMAGE_IDENTIFIER`: exact pinned scientific/HTTP image ID used only for no-network preflight. `PULSATE_IBM_IMAGE_IDENTIFIER` is the distinct exact IBM Runtime image ID used only by the network phase.

Local simulator execution requires no secret and does require the exact packages in `requirements/quantum-preflight.lock`. Missing or mismatched dependencies fail truthfully; there is no fixture-energy fallback.

## Pinned Linux HTTP integration image

The trusted `cgr-quantum-preflight:1.0.0` scientific base image and
`requirements/quantum-preflight.lock` remain unchanged. The derived
`cgr-pulsate-http-integration:1.0.0` image adds the FastAPI, HTTPX, and
Uvicorn top-level application dependencies and their hash-locked transitive
dependencies, including Starlette, as defined by
`requirements/pulsate-http-integration.lock`.

On a Linux Docker host, build both layers, print their immutable image IDs,
verify the combined package environment, and run the real HTTP integration with:

```bash
bash ./scripts/run-pulsate-http-integration.sh
```

Runtime networking remains disabled, the filesystem is read-only except for the
bounded `/tmp` mount, and the container runs as the inherited unprivileged user.
`PULSATE_QUANTUM_IMAGE_IDENTIFIER` receives the derived image ID because that is
the actual combined execution environment.

This is a local H2 simulator integration through the existing scientific runner;
it is not IBM Quantum execution. There is no fixture-energy fallback, Qwen call,
natural-language compilation, or alternate authorization path.

## IBM Quantum Execution v1

An explicit IBM request retains `execution_target=ibm_quantum`. It becomes
executable only when the server has the pinned `qiskit-ibm-runtime` client and
token, instance, and backend environment variables. The browser never supplies
credentials, backend objects, precision, or transpiler options.

The existing isolated exact/VQE workflow runs first in a separate no-network
process or container and must authorize. After the API allocates its random run
identifier, it atomically publishes a typed request to a live no-network
coordinator. Capability requires that coordinator's fresh readiness heartbeat;
an empty exchange directory is insufficient. The coordinator runs trusted
preflight for that exact run and atomically publishes a run-bound,
cryptographically checked handoff. The API waits for and validates that
handoff; it never falls back to local preflight in the network-capable API
process. Pulsate then binds the
locally optimized UCCSD vector, experiment, Mol* structure,
Hamiltonian, circuit, and observable fingerprints into a bounded internal
submission. A separate network-enabled worker reconstructs that evidence,
transpiles the bound circuit for the configured backend, applies the layout to
the observable, and submits one EstimatorV2 expectation evaluation. Its truthful
hardware role is `final_energy_evaluation_at_locally_optimized_parameters`; it is
not hardware-optimized VQE.

Before an attempt, the worker atomically persists the exact prepared ISA
workload as bounded QPY circuit bytes, a canonical laid-out observable, and a
prepared-evidence envelope. The envelope binds the immutable bundle, source and
transpiled identities, physical layout, backend target identity when available,
Qiskit version, optimization level, and server-owned transpiler seed. Recovery
loads those exact files and never retranspiles.

The worker records an atomic submission attempt before `EstimatorV2.run()`.
The attempt contains the immutable bundle identity and a deterministic IBM job
tag. Once an attempt exists, recovery never blindly resubmits: zero matching
jobs remains indeterminate, one match is recorded and retrieved, and multiple
matches fail as a duplicate-submission integrity error. `job.json` is written
immediately when the job identifier becomes known.

Post-submission service, status, and result-retrieval failures cross the worker
boundary only as a bounded typed failure envelope. It contains a controlled
category, persisted-job flag, optional job identifier, last controlled status,
and retrieval disposition. Provider text, credentials, and paths are excluded.
Recoverable jobs remain restartable; terminal cancelled/error/failed jobs retain
their public job/backend state.

Runtime status is projected only from actual `RuntimeJobV2` polling. The
immutable bundle's server-controlled maximum is applied to
`EstimatorV2.options.max_execution_time`; the local retrieval grace is separate.
Raw mapped-Hamiltonian expectation, non-nuclear electronic constants,
electronic energy, nuclear repulsion, and total energy are retained separately,
so nuclear repulsion is added exactly once.

Canonical bounded QPY bytes identify the bound source and transpiled circuits.
Canonical `SparsePauliOp` serialization identifies both source and
layout-applied observables; the receipt also binds the physical layout identity.
Public evidence includes the job/backend and scientific identities but excludes
credentials and filesystem paths. Execution-integrity and noisy-hardware quality
are separate decisions; the IBM quality thresholds are server policy rather than
the local simulator's 1e-5 Hartree agreement threshold.

The dedicated `cgr-pulsate-ibm-runtime:1.0.0` image derives from the exact pinned
HTTP image and installs `requirements/pulsate-ibm-runtime.lock`. Live validation
uses a persistent controlled Docker volume and two execution containers. The
exact scientific/HTTP image runs PySCF/Qiskit preflight credential-free under
`--network none`; only the distinct derived IBM Runtime image runs the network
phase. The handoff and public IBM receipt retain
`scientific_preflight_image_identifier` and `ibm_runtime_image_identifier`
separately. Missing, unknown, unpinned, local, or equal identities fail closed.
Live validation requires all cost gates and credentials:

```bash
bash ./scripts/run-pulsate-ibm-integration.sh
```

The separate non-paid orchestration check uses an internal Docker network and a
network-local fake adapter endpoint. It unsets IBM credentials and cost gates,
runs local preflight with `--network none`, exercises HTTP plan/run and restart
retrieval, proves at-most-once fake submission, and checks receipt/Mol*
structure identity:

```bash
bash ./scripts/run-pulsate-ibm-fake-integration.sh
```
