# Pulsate preset run API

## Scope

The versioned Pulsate API coordinates a discovered quantum-preflight manifest through the repository's existing trusted runner. It does not reimplement problem construction, Hamiltonian construction, exact diagonalization, VQE, verification, authorization, or receipt generation. `ExistingQuantumPreflightExecutor` calls `run_trusted_reference` and projects its canonical artifacts into typed browser responses.

This is a bounded, single-process development coordinator, not a production-distributed queue. This path does not compile natural language, invoke Qwen repair, or use IBM Quantum Runtime.

```text
discovered preset -> trusted ManifestEnvelope -> durable run envelope
  -> ExistingQuantumPreflightExecutor -> authoritative runner artifacts
  -> typed results, verification, and receipt responses
```

Preset, experiment, structure, Hamiltonian, run, receipt, and execution-environment identities remain separate. Internal filesystem paths are never returned.

## API and status

- `GET /api/v1/health` and `GET /api/v1/runs/capability` expose truthful local capability.
- `POST /api/v1/runs` accepts only `{ "preset_identifier": "...", "execution_target": "local_simulator" }` and returns `202`.
- `GET /api/v1/runs/{run_identifier}` returns durable status and identity.
- `/results`, `/verification`, and `/receipt` return strict typed projections of authoritative evidence. The public receipt exposes stable identities and authorization, while the complete canonical receipt remains server-controlled evidence. Before evidence exists they return `409` with a typed error.

The browser cannot supply paths, commands, executables, arguments, result roots, images, or code. Unknown fields are rejected. Run identifiers and directories are generated by the server. An `Idempotency-Key` of 8–128 identifier characters deduplicates an identical request; conflicting reuse returns `409`.

Only observable states are published: `queued`, `validating`, `running_quantum_workflow`, `authorized`, `rejected`, `failed`, and `interrupted`. The last four are terminal. Rejection is a completed scientific decision; failure is an execution or artifact-processing error.

## Persistence and recovery

The default root is ignored `.pulsate-runs/`. Each server-named run directory contains atomic request/state JSON and the required summary, results, verification, receipt, or sanitized error. The nested quantum-preflight artifact directory remains authoritative.

Coordinator construction and module import do not create directories, inspect quantum dependencies, recover runs, acquire resources, or start worker threads. FastAPI lifespan explicitly starts and closes the coordinator. Startup acquires an exclusive run-root ownership lock, validates configuration and execution prerequisites, and only then recovers persisted state. A second live coordinator cannot own the same run root.

On startup, terminal evidence is preserved, idempotency records are restored, and active runs become `interrupted`; calculations are never silently restarted or deleted. Shutdown stops admission before waiting for bounded workers and releasing root ownership. State is lock protected and JSON replacement is atomic. Concurrency defaults to one and is clamped to four.

Before projection, every required runner artifact must be a regular, non-symlink file directly inside the controlled result directory and within the manifest result-size policy. The receipt, result artifacts, verification report, warnings, manifest, summary, and authorization decisions are model-validated and cross-linked by recomputed identities. Inconsistent evidence fails the run and cannot authorize through the browser API.

## Configuration and development

- `PULSATE_EXECUTION_ENABLED`: `true` enables local execution; default `false`.
- `PULSATE_RUN_ROOT`: durable root; default `<repository>/.pulsate-runs`.
- `PULSATE_MAX_CONCURRENT_RUNS`: 1–4 workers; default `1`.
- `PULSATE_MAX_RUN_SECONDS`: server execution ceiling from `1` through `3600`; default `180`. The trusted runner receives the lower of this value and the manifest execution-policy limit.
- `PULSATE_QUANTUM_IMAGE_IDENTIFIER`: environment evidence identifier; default `local-uncontainerized`.

Local simulator execution requires no secret and does require the exact packages in `requirements/quantum-preflight.lock`. Missing or mismatched dependencies fail truthfully; there is no fixture-energy fallback.

```powershell
$env:PYTHONPATH = "$PWD\src"
$env:PULSATE_EXECUTION_ENABLED = "true"
python -m uvicorn cgr.pulsate_api.app:app --host 127.0.0.1 --port 8000
```

On the existing Linux EC2 Docker host, rebuild the trusted image and run the opt-in real HTTP integration exactly as follows:

```bash
./scripts/build-quantum-preflight-image.sh
image_id="$(docker image inspect --format '{{.Id}}' cgr-quantum-preflight:1.0.0)"
docker run --rm --network none --read-only --cpus 2 --memory 4g --pids-limit 256 \
  --security-opt no-new-privileges --cap-drop ALL \
  --tmpfs /tmp:rw,nosuid,nodev,size=1g,mode=1777 \
  --env PULSATE_RUN_HTTP_INTEGRATION=true \
  --env PULSATE_EXECUTION_ENABLED=true \
  --env PULSATE_RUN_ROOT=/tmp/pulsate-runs \
  --env "PULSATE_QUANTUM_IMAGE_IDENTIFIER=$image_id" \
  --entrypoint python cgr-quantum-preflight:1.0.0 \
  -m pytest -q -s -rs -p no:cacheprovider tests/test_pulsate_runs_integration.py
```

This command uses the image's pinned packages, disables networking, runs the real existing runner through the HTTP API, and fails unless the run is authorized with readable results, verification, and receipt evidence.
